## Overview

This document defines quality standards for code generated by react2htmx. The tool converts Figma Make React applications into Buckler-compatible HTMX/FastAPI modules. Generated code must be production-ready, maintainable, and follow Buckler conventions.

**Core Principle**: Generated code should be indistinguishable from hand-written code by an expert developer.

---

## 1. DATAOS Compliance (DOM as Authority on State)

### 1.1 State Classification Requirements

All generated code MUST follow DATAOS principles for state management:

| React Pattern | Classification | Generated Target |
|---------------|----------------|------------------|
| `useState` (UI: open/closed, selected) | UI State | DOM attributes + domx manifest |
| `useState` (form values) | Form State | DOM values (read on submit) |
| `useState` (loading, hover) | Ephemeral | CSS classes / HTMX indicators |
| User session data | Session State | Redis (server-side) |
| Fetched data, records | Persistent | PostgreSQL (server-side) |

### 1.2 Detection Heuristics

When classifying React state, apply these rules:

```
IF state used only for rendering → DOM (domx manifest)
IF state sent to API / persisted → Server (Redis or DB)
IF state survives page refresh → Server
IF state loss on refresh is acceptable → DOM
```

### 1.3 Generated domx Manifest

For UI state, generate domx manifests:

```javascript
// Generated: static/js/domx-manifest.js
const domxManifest = {
  "accordion-panel-1": {
    attribute: "data-expanded",
    default: "false",
    persist: false
  },
  "dropdown-user-menu": {
    attribute: "data-open",
    default: "false",
    persist: false
  },
  "selected-tab": {
    attribute: "data-active-tab",
    default: "overview",
    persist: true  // Survives page refresh via replay()
  }
};
```

## 2. FastAPI Route Generation

### 2.1 Route Handler Structure

```python
# Generated: routes/auth.py
from fastapi import APIRouter, Request, Depends, HTTPException
from fastapi.responses import HTMLResponse
from sqlalchemy.orm import Session

from flanker_notifications import NotificationService
from ..domain.auth import LoginRequest, LoginResponse
from ..services.auth import authenticate_user
from ..dependencies import get_db, get_templates

router = APIRouter(prefix="/auth", tags=["auth"])
notification_service = NotificationService()
templates = get_templates()


@router.post("/login", response_class=HTMLResponse)
async def login(
    request: Request,
    data: LoginRequest,
    db: Session = Depends(get_db),
) -> HTMLResponse:
    """Handle user login.

    Returns dashboard partial on success, login form with errors on failure.
    """
    result = authenticate_user(data.email, data.password, db)

    if result.success:
        notification_service.success("Welcome back!")
        return templates.TemplateResponse(
            request=request,
            name="partials/dashboard.html",
            context={"user": result.user},
        )

    notification_service.error(result.message, priority="informational")
    return templates.TemplateResponse(
        request=request,
        name="partials/login-form.html",
        context={"errors": {"credentials": result.message}},
    )
```

### 2.2 Route Generation Rules

**MANDATORY**:
- Use `APIRouter` with appropriate prefix and tags
- Include `response_class=HTMLResponse` for HTMX endpoints
- Use dependency injection for database sessions
- Return `TemplateResponse` for partial rendering
- Integrate `flanker_notifications` for user feedback

**FORBIDDEN**:
- Raw `hx-swap-oob` without notification service
- JSON responses for HTMX endpoints (use partials)
- Direct database access in route handlers
- Business logic in route handlers (delegate to services)

### 2.3 Security Decorators

Generate appropriate security decorators based on detected patterns:

```python
from buckler.security import requires_auth, requires_role, rate_limit

@router.post("/admin/users")
@requires_auth
@requires_role("admin")
@rate_limit(requests=10, window=60)
async def create_user(
    request: Request,
    data: CreateUserRequest,
    db: Session = Depends(get_db),
) -> HTMLResponse:
    """Create new user (admin only)."""
    ...
```

### 3.4 HTMX Attribute Mapping

| React Pattern | HTMX Equivalent |
|---------------|-----------------|
| `onClick` → fetch | `hx-get` / `hx-post` |
| `onSubmit` → API call | `hx-post` on `<form>` |
| `useEffect` + interval | `hx-trigger="every Xs"` |
| Conditional render | `{% if condition %}` |
| `.map()` render | `{% for item in items %}` |
| Loading state | `hx-indicator` |
| Optimistic update | `hx-swap="outerHTML"` with server response |

### 3.5 Template Generation Rules

**MANDATORY**:
- Include CSRF tokens in all forms
- Use `hx-indicator` for loading states
- Use semantic HTML elements
- Include error display blocks
- Use Jinja2 url_for() for all URLs

**FORBIDDEN**:
- Inline JavaScript for behavior (use HTMX)
- Inline styles (use CSS classes)
- Hard-coded URLs
- Missing form validation feedback

## 4. CSS Generation

### 4.1 Class Naming Hierarchy

Generated class names follow this priority order:

1. **Figma-derived** (preferred): Component name + prop context
   ```css
   .login-form__submit-button { }
   .checkout__cancel-action { }
   .user-profile__avatar { }
   ```

2. **LLM-inferred**: Screen context + element role
   ```css
   .hero-section__cta { }
   .pricing-table__highlight { }
   ```

3. **Mechanical fallback**: DOM position (least preferred)
   ```css
   .section-3__div-2 { }
   .container-1__element-5 { }
   ```

**MANDATORY**:

**FORBIDDEN**:
- Tailwind utility classes in output (convert to semantic classes)
- `!important` declarations
- ID selectors for styling
- Vendor prefixes without autoprefixer
- Magic numbers (use CSS custom properties)

## 5. Domain Model Generation

### 5.1 Pydantic Model Structure

```python
# Generated: domain/auth.py
from pydantic import BaseModel, EmailStr, Field, field_validator
from typing import Optional


class LoginRequest(BaseModel):
    """Request model for user login."""

    email: EmailStr = Field(
        ...,
        description="User's email address",
        examples=["user@example.com"],
    )
    password: str = Field(
        ...,
        min_length=8,
        description="User's password",
    )

    @field_validator("password")
    @classmethod
    def password_not_empty(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("Password cannot be empty")
        return v


class LoginResponse(BaseModel):
    """Response model for login attempt."""

    success: bool
    message: Optional[str] = None
    user: Optional["UserSummary"] = None


class UserSummary(BaseModel):
    """Summary of user data for responses."""

    id: str
    email: EmailStr
    display_name: str
```

### 5.2 Model Generation Rules

**MANDATORY**:
- Derive models from Gherkin scenarios
- Include Field descriptions and examples
- Add appropriate validators
- Use strict types (EmailStr, HttpUrl, etc.)

**FORBIDDEN**:
- Generic `dict` types for structured data
- Missing field descriptions
- Unvalidated string inputs for sensitive data

## 7. Gherkin Feature Generation

### 7.1 Feature File Structure

```gherkin
# Generated: tests/features/user_login.feature
Feature: User Login
  As a registered user
  I want to log in to my account
  So that I can access protected features

  Background:
    Given the login page is displayed
```

### 7.2 Gherkin Generation Rules

**MANDATORY**:
- One feature file per functional unit
- Include Background for common setup
- Cover happy path scenario first
- Include error/edge case scenarios
- Use Buckler's pytest-BDD step patterns

**Functional Units to Detect**:
- Forms (inputs + submit handlers)
- Data displays (lists, tables, cards)
- Navigation (links, tabs, modals)
- State toggles (dropdowns, accordions)
- Async operations (loading states)

## 8. Notification Integration

### 8.1 flanker-notifications Usage

```python
from flanker_notifications import NotificationService

notification_service = NotificationService()

# Success notification (toast, auto-dismiss)
notification_service.success("Changes saved successfully")

# Error notification (informational priority)
notification_service.error("Invalid input", priority="informational")

# Warning notification
notification_service.warning("Session will expire in 5 minutes")

# Confirmation modal (blocking)
response = await notification_service.confirm_modal(
    "Delete this item?",
    buttons=["Delete", "Cancel"],
    default_button="Cancel",
)
if response["action"] == "Delete":
    # proceed with deletion
```

### 8.2 Notification Zone Mapping

| Notification Type | Zone | Behavior |
|-------------------|------|----------|
| Success | `toast_corner` | Auto-dismiss after 3s |
| Info | `toast_corner` | Auto-dismiss after 5s |
| Warning | `banner_top` | Persistent until dismissed |
| Error (informational) | `toast_corner` | Auto-dismiss after 5s |
| Error (critical) | `modal_center` | Requires acknowledgment |
| Confirmation | `modal_center` | Blocks until response |

## 9. Partial Template Sizing

### 9.1 Line Count Guidelines

| Line Count | Action |
|------------|--------|
| < 500 lines | Merge with parent or siblings |
| 500-1200 lines | Optimal - keep as-is |
| > 1200 lines | Split into sub-partials |

**Target**: ~750 lines per partial template

### 9.2 Merge Candidates

- Icon components
- Simple wrapper/layout components
- Components that only pass props through

### 9.3 Split Candidates

- Components with multiple distinct sections
- Components with independent data fetching
- Components that could load progressively

## 12. Conclusion

Generated code quality is the primary measure of react2htmx success. Code that requires extensive manual cleanup defeats the tool's purpose.

**Quality Bar**: A developer should be able to use generated code immediately, needing only to:
1. Implement service function bodies
2. Wire up database models
3. Add security decorators
4. Refine CSS class names (if mechanical naming was used)

Everything else should be production-ready out of the box.
